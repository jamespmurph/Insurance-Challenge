# What We Look For — Apprentice

This guide describes the qualities and behaviours we are assessing, beyond whether the code works.

---

## Core technical ability

We want to see that the candidate can:

- Read and understand existing code they did not write
- Follow established patterns in the codebase (e.g. how `PolicyHandler` is structured)
- Write basic C# with correct syntax and conventions
- Implement validation logic that guards against bad input
- Connect components correctly (handler → repository)

---

## Problem solving

We are looking for candidates who:

- Start with the simplest working solution and build from there
- Can reason about what needs to happen even when it is not fully spelled out
- Do not give up when something is not immediately obvious

---

## Communication

`COMPLETION.MD` gives us a window into how the candidate thinks. We look for:

- A clear, honest account of what they did and why
- Acknowledgement of anything they found difficult or would revisit
- Concise writing — this does not need to be long

---

## What we do not penalise

- Not knowing every API or syntax from memory — looking things up is fine
- Imperfect code that still shows understanding of the problem
- Short `COMPLETION.MD` entries, as long as they are honest

---

## Signals of a strong apprentice candidate

- They implement the full claim flow end-to-end, including `Program.cs`
- Their validation error messages are clear and specific
- They explain their decisions in `COMPLETION.MD` with confidence
- They go slightly beyond the minimum (e.g. handle an edge case they noticed)
